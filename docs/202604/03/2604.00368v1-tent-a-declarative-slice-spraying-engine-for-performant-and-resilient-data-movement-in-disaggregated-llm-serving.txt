## Abstract 

Modern GPU clusters are built upon a complex hierarchy of heterogeneous interconnects, ranging from multi-rail RDMA to proprietary fabrics such as Multi-Node NVLink and As-cend UB. Orchestrating these diverse links effectively remains a critical challenge in disaggregated LLM serving. Operat-ing Mooncake TE on thousands of GPUs exposed a critical limitation shared by existing frameworks: imperative, stati-cally bound path selection . This rigidity forces engines to rely on state-blind striping that ignores congestion signals, creat-ing communication silos, wasting multi-rail bandwidth due to head-of-line blocking, and leading to operational fragility where routine faults require manual intervention. We present TENT , a data-movement engine that decouples transfer intent from physical execution . Instead of locking workloads to fixed backends, TENT unifies heterogeneous interconnects into a single dynamic resource pool. Applica-tions simply declare transfer intents, while TENT dynami-cally decomposes elephant flows into fine-grained slices and “sprays” them across links based on instantaneous link quality .This telemetry-driven orchestration eliminates head-of-line blocking and enables transparent, sub-50 ms self-healing by rerouting slices around failures without application logic. TENT serves as the production data plane for LLM in-ference and RL pipelines at multiple industrial sites. Our evaluation on H800 HGX clusters shows that TENT out-performs state-of-the-art baselines, including Mooncake TE, NIXL, and UCCL. In LLM inference with SGLang HiCache, TENT achieves up to 1.36 × higher throughput and 26% lower P90 TTFT than Mooncake TE. In RL pipelines, TENT accel-erates parameter updates in Moonshot Checkpoint Engine by 20 ∼ 26%.

These links differ in band-width, latency, and control plane, accumulating mixed - vendor and mixed -fabric deployments. In this setting, the dominant challenge of the data plane has shifted from achieving high peak bandwidth in isolation to coordinating all available links into a single, resilient, and programmable data-movement sub-strate for latency-sensitive, high-volume flows. It must choose transport protocols, decide how to stripe large transfers across multiple rails, and when to orchestrate compound paths that relay data through intermediate devices, etc. This challenge is especially acute in disaggregated LLM serving, where high -volume, latency -critical “ elephant flows ”lie directly on the serving critical path. In these scenarios, transferring gigabytes of tensors or model parameters between GPUs and hosts dominates end - to -end latency. Production systems therefore rely heavily on advanced point - to -point (P2P) transfer engines such as Mooncake Transfer Engine (TE) [25], NVIDIA NIXL [21], UCCL [39] and DLSlime [5]. However, operating Mooncake TE across heterogeneous clusters revealed that the imperative path-selection model, while effective in homogeneous setups, lacks the flexibility required to fully exploit thousands of mixed-generation GPUs. By forcing applications to explicitly bind to specific transport backends (e.g., NVLink, GDS, or ROCm) at initialization or compilation, these systems tightly couple workloads to hardware. Static binding creates communication silos. Each deploy-ment binds to a specific transport stack at initialization, mak-ing clusters compiled with different transport backends unable to interoperate. TENT makes a paradigm shift from imperative path selection to declarative slice spraying : Applications declare what data to move (seg-ments and endpoints) via a familiar BatchTransfer -style API. TENT decides how to realize each transfer across all available interconnects, continuously adapting choices to run-time conditions. Concretely, TENT makes three key shifts: • From Static Binding to Dynamic Orchestration. 

TENT unifies heterogeneous interconnects (memory-direct, storage-direct, and proprietary fabrics) under a single segment abstraction (host memory, accelerator memory, and persistent storage). Instead of Mooncake TE’s committing to a specific transport backend at ini-tialization, TENT performs late binding at runtime. It maintains a dynamic reachability map and routes trans-fers on a per-request basis, allowing it to adapt to topol-ogy changes and utilize any reachable path dynamically (e.g., transparently synthesizing a staged route via host memory when direct P2P is unavailable). • From State-Blind Striping to Telemetry-Driven Slice Spraying. To overcome the head-of-line blocking in-herent in static striping, TENT decomposes elephant flows into fine-grained slices (e.g., 64 KB) and sched-ules them based on instantaneous link state. Rather than fixed priorities, it directs slices to rails with the lowest estimated completion time (derived from bandwidth and queued bytes). This continuous feedback loop enforces a declarative slice-level policy that maximizes multi-rail utilization, keeps tail latency low, and supports band-width spreading when multiple instances share the same NICs. • From Brittle Execution to Resilient Self-Healing. 

TENT moves error handling from the application into the data plane. It treats transport faults not as excep-tions, but as routine routing events: if a slice fails (due to link flapping or timeout), the runtime automatically re-schedules it on an alternative path in sub-milliseconds. This in-band recovery masks transient hardware failures, ensuring reliable delivery without requiring application-level retries or checkpoints. TENT has been deployed as the common data -movement substrate for both inference and reinforcement-learning pipelines in large industrial environments, serving billions of tokens per day on clusters with up to thousands of GPUs. In a leading financial technology company, TENT has been deployed as the enterprise-wide LLM inference infrastructure. It provides explicit SLA guarantees, consistently achieving sub-second time-to-first-token (TTFT) and sub-30 ms time-per-output-token (TPOT) in production. ## 2 Operational Case Studies with Mooncake TE in Heterogeneous Clusters 

As Mooncake TE became the default transfer engine within our LLM serving stack, its deployment expanded to encom-pass thousands of GPUs spanning multiple generations, ven-dors, and interconnect technologies. Modern servers in this fleet integrate a complex hierarchy of fabrics: dedicated GPU-to-GPU links (e.g., NVLink [22], AMD Infinity Fab-ric [3], Ascend UB [14]), CPU-bypass paths (e.g., GPUDirect Storage [18]), and rack-scale fabrics (including Multi-Node NVLink (MNNVL) and multi-rail RDMA), as shown in Fig-ure 1. These links exhibit sharp disparities in bandwidth, la-tency, reachability, and NUMA affinity. Furthermore, as clus-ters increasingly mix accelerator generations, not all memory regions are equally reachable from every fabric. Existing engines (e.g., Mooncake TE [25], NIXL [21], UCCL [39], and DLSlime [5]) were designed for relatively homogeneous environments, adhering to an imperative path-selection model where applications commit to a fixed backend stack at compilation or initialization. As hardware diversity increased, the rigidity of this model manifested as significant operational bottlenecks. This section details three representa-tive challenges that motivated the design of TENT. 

## 2.1 Static Binding in a Mixed-vendor Fleet 

The first operational challenge surfaced as communication silos that fragmented the logical cluster. From an operator’s perspective, services within the same LLM stack could not seamlessly share capacity or route around unhealthy compo-nents. This fragmentation was exacerbated by the deployment of specialized fabrics available only on specific rack subsets: GPUDirect RDMA and NVLink are exclusive to NVIDIA dat-acenter GPUs; Multi-Node NVLink (MNNVL) is restricted to systems such as the GB200 NVL72; and HIXL is supported only on Huawei Ascend devices. Even within a single data-center, connectivity profiles varied: MNNVL is optimized for GPU-to-GPU transfers and cannot handle host-to-host paths, while many older GPUs lack basic P2P support entirely. Mooncake TE mirrored this hardware heterogeneity in the software layer. This forced us to maintain separate container images and configurations for each environ-ment, e.g., “NVLink+RDMA”, “Ascend HIXL+RoCE”, and “legacy TCP-only”. While the cost of this design remained hidden in homogeneous clusters, it became a major bottleneck as heterogeneity increased. Once a service was statically bound to a backend, its trans-fers could not opportunistically combine or switch fabrics, even when different segments of a flow would benefit from different links. For instance, optimally serving a request might require using RDMA for host-to-GPU segments while simul-taneously exploiting MNNVL for GPU-to-GPU hops. Under the static model, achieving this required manual staging and complex logic, rather than emerging naturally from the en-gine’s path selection. The same rigidity prevented the runtime from reclaiming capacity on newly provisioned fabrics or transparently steering around temporarily degraded ones. Operationally, this resulted in backend-specific runbooks, deployment pipelines, and dashboards, forcing operators to reason about capacity and resilience per isolated island rather than at the scale of the entire fleet. The core symptom that services failed to fully exploit available interconnects or adapt to hardware changes, was a direct consequence of treating backend selection as a static initialization decision. TENT addresses this by shifting path selection to a runtime orches-tration layer, maintaining a global topology view to decouple workload binaries from the exact hardware configuration be-neath them. 

## 2.2 Multi-rail Striping for Non-uniform Fabric 

The second operational challenge concerns the tail latency of large, critical-path transfers. As LLM serving has evolved from small, monolithic deployments to large-scale, disaggre-gated production systems, workloads are increasingly domi-nated by elephant flows that directly impact user experience. For instance, prefilling a single 1024-token prompt with the DeepSeek-R1 model generates 1.668 GB of KVCache tensors, and retrieving remote KVCaches routinely involves moving hundreds of megabytes per request. In principle, multi-rail server configurations should accelerate these flows linearly; in practice, operational telemetry revealed aggregate throughput well below the sum of per-rail capacities, along with skewed NIC utilization and sharp tail latency spikes during check-pointing and cache retrieval. 3These symptoms stem directly from the state-blind striping policies employed by existing engines. Both Mooncake TE and UCX attempt to exploit multi-rail bandwidth by decom-posing large transfers into slices. However, their distribution logic ignores instantaneous link health: UCX relies on static heuristics and coarse-grained segmentation, while Mooncake TE distributes fixed-size 64 KB chunks using round-robin or hashing based solely on static NUMA priorities. In production, however, rail performance is highly non-uniform and time-varying. Some rails are physically distant from submission threads (cross-NUMA); others contend with noisy neighbors or suffer from transient signal degradation that reduces effective bandwidth without triggering hard fail-ures. When slices of an elephant flow are assigned to such a degraded rail, they become stragglers , forcing the entire transfer to stall. This manifests as classic Head-of-Line (HoL) blocking: end-to-end latency is constrained by the slowest path rather than defined by the aggregate fabric bandwidth. We reproduced this behavior experimentally on an eight-rail 200 Gbps RDMA fabric. We issued read requests that split into 1 MB slices, from four submission threads capable of posting to any NIC. We compared a baseline round-robin dispatcher against the telemetry-driven scheduler adopted by TENT (§4.2). Under round-robin, rails attached to remote NUMA domains exhibited significantly higher per-slice ser-vice times; the resulting queue buildup on those specific rails inflated P99 latency for the entire request, as shown in Fig-ure 2. In contrast, when the scheduler utilized live teleme-try (queued bytes and completion latency) to steer slices away from backlogged rails, throughput increased, tail la-tency dropped, and progress remained balanced across the fabric. Thus, on the same heterogeneous fabric that exposed static binding issues, state-blind striping turned physical non-uniformity into performance cliffs. TENT addresses this by treating all rails as a unified resource pool, decomposing trans-fers into fine-grained slices, and scheduling each slice dynam-ically based on a telemetry-aware cost model.

The third class of issues arose not from steady-state perfor-mance but from ordinary hardware churn in large datacenters. Even in well-maintained clusters, components fail and re-cover continuously. A clear pattern emerged: GPU-side disturbances dominated the failure surface, with soft errors, thermal excur-sions, and intermittent device dropouts frequently cascading into communication disruptions. At the level of the transfer engine, these faults rarely present as a clean node down . Instead, they appear as abrupt changes in path availability or quality. In Mooncake TE’s design, a transfer bound to a particular back-end and path failed if that path became unusable, and recovery was delegated to orchestration systems and on-call operators; the data plane could neither reroute in-flight slices around a degraded path nor reintegrate a repaired rail automatically. Worse, when hardware recovered quickly, services often remained in a degraded configuration: jobs that had fallen back from a fast GPU–GPU fabric to a slower host-mediated path tended to stay there until an operator forced a restart or reconfiguration, because nothing in the data plane actively probed and exploited the restored fabric. This failure-handling model proved inadequate for disag-gregated LLM serving. Small, localized glitches could stall critical paths or silently depress capacity long after the un-derlying problems had been repaired. Together with the first two case studies, this completes the picture: static binding fragmented the fabric, state-blind striping squandered multi-rail potential, and control-plane-only recovery turned routine churn into major operational events. ## 3 TENT Architecture 

Mooncake TE exposes a batch transfer interface in which applications register local buffers, allocate batch identifiers, submit groups of transfers, and later query batch completion. Large KVCache blocks are sliced and sent over RDMA, typ-ically across multiple NICs, using topology-aware policies that achieve high throughput and low cross-node latency in relatively homogeneous RDMA clusters. As discussed in §2, large-scale LLM serving increasingly spans heterogeneous memory tiers and interconnects, mak-ing TE’s static-binding assumptions insufficient for mod-ern deployments. TENT addresses these challenges by re-architecting the transfer engine around two core abstractions: 

pluggable transport backends that unify heterogeneous trans-port mechanisms, and segments that normalize data location and representation. Built on top of these abstractions, TENT applies a suite of runtime scheduling and resilience mech-anisms that dynamically adapt data movement to changing hardware capabilities and system conditions. Figure 3 provides an overview of TENT’s architecture. At a high level, applications describe what data should be moved through a declarative, batch-oriented API, while TENT de-cides how to realize each transfer over the available fabric. Building on this structure, TENT organizes data movement around two core abstractions: segments (§3.1), which provide a unified way to refer to data, and pluggable transports (§3.2), which integrate diverse interconnects under a common execu-tion interface. ## 3.1 Segment Representation 

In heterogeneous clusters, both data placement and access mechanisms vary across devices and platforms. Without a unified abstraction, applications must reason explicitly about where data resides and which transport to use, leading to rigid coupling and communication silos. TENT eliminates this complexity by introducing the segment abstraction (Figure 4). A segment represents a logical data region mapped to one or more contiguous buffers , independent of the underlying storage or memory medium. Segments unify data residing in host DRAMs, GPU HBMs, SSDs, and NVMe-oF targets into a single, transport-agnostic namespace. Applications inter-act exclusively with segment identifiers, offsets, and lengths, without exposing transport- or device-specific details. When submitting transfer requests, an application specifies a batched transfer between source and destination segments. Crucially, these requests encode only logical intent— move data from segment X to segment Y —and contain no trans-port bindings. This strict separation decouples application logic from hardware capabilities, allowing TENT to defer all placement and routing decisions to runtime and to expose a unified completion model across heterogeneous transports, simplifying application-side control flow. Internally, each segment encapsulates device-specific meta-data, such as RDMA keys or GPU memory handles, within a normalized structure. These details are opaque to the core en-gine and accessed only by the appropriate backend, ensuring that new storage or memory technologies can be integrated without modifying scheduling logic or application interfaces. 

Building Segment Metadata. At initialization, TENT per-forms automated topology discovery, enumerating NICs, GPUs, storage devices, and their interconnects. These links are then classified into protocol-independent affinity tiers (Fig-ure 4): tier-1 for optimal paths such as NVLink or GPUDi-rect RDMA, tier-2 for cross-root connections, and tier-3 

for NUMA-crossing fallbacks. The resulting tiered topology graph, incorporated into each segment’s metadata, serves as the global ground truth for routing. Using this topology from metadata, TENT derives the trans-5port capabilities of each segment and records them in its meta-data, enabling the orchestrator to determine feasibility and affinity without examining device-specific details. Through-out a segment’s lifecycle, the segment manager maintains this metadata and retrieves remote metadata on demand, ensuring that the orchestrator always has an up-to-date view of where data resides and which transports remain available. 

## 3.2 Pluggable Transport Backends 

To accommodate diverse interconnects, TENT defines a uni-form transport backend interface that abstracts physical data movement. Each fabric (for example RDMA, NVLink, MN-NVL, Ascend HIXL, TCP, shared memory, or file backed I/O) is implemented as a backend conforming to this interface and loaded dynamically at runtime. In contrast to Mooncake TE, where each transport main-tained its own segment metadata format and therefore con-strained an instance to a single transport, TENT generalizes metadata representation while maintaining transport-specific fields (Figure 4). This design isolates transport specific mech-anisms from the core engine and allows new fabrics to be integrated simply by implementing the backend interface and declaring their capabilities (such as peer accessibility, device locality, or supported transfer directions). Once registered, backends are incorporated uniformly into TENT’s scheduling and resilience logic without any changes to higher layers. Most backends are intentionally lightweight. They typically encapsulate only the minimal initialization steps and a small number of data movement primitives. This modularity enables rapid support for emerging interconnects and vendor specific fabrics while maintaining a unified execution framework. 

## 3.3 Execution Flow Overview 

We next summarize how TENT’s architectural components in-teract during the execution of a transfer request, as illustrated in Figure 3. 

Initialization. Before any transfer occurs, TENT discovers the hardware topology and loads all available transport back-ends. During memory registration, the engine constructs seg-ment metadata using the discovered topology (§3.1) and stores it for later use by both the orchestrator and the backends. These steps populate the global state on which subsequent execution depends. 

Control Flow. The control flow governs what actions should execute and when, while leaving all data movement to the data plane. It begins when an application submits a batched transfer specifying logical source and destination segments, offsets, and lengths. The request contains only intent and binds to no specific transport. The core engine forwards this request to the runtime orchestrator, which serves as the control plane. Guided by global topology and backend capabilities, the orchestrator resolves the request into an execution plan that enumerates feasible paths through the heterogeneous fabric. Large transfers are divided into slices, forming the units of scheduling and isolation. Multiple candidate routes and backends are retained so that binding can be deferred and chosen dynamically. 

Data Flow. The data flow performs the actual movement of bytes and never makes routing or scheduling decisions. Once the orchestrator dispatches a slice, a pluggable transport backend executes it using a uniform slice-execution interface, transferring data along the selected path. Backends report completions and telemetry to the orchestrator, which applies uniform scheduling and resilience policies across all trans-ports. Because all backends conform to the same interface, the orchestrator can treat heterogeneous interconnects as in-terchangeable execution resources. 

Failure Handling. The failure-handling loop captures how TENT reacts to errors during execution. If a backend reports a failure or degradation, the orchestrator retries or redirects the affected slice using the remaining candidates from the execution plan. A batched transfer completes once all slices finish, at which point the engine emits a single completion event to the application. These adaptations occur transparently and do not change the API exposed to upper-layer serving and reinforcement-learning pipelines. 

## 4 TENT Execution Pipeline 

Based on the above design, TENT proposes the execution pipeline with three phases: dynamic orchestration (§4.1) chooses paths at request time, telemetry-driven slice spray-ing (§4.2) distributes large transfers across healthy rails, and dual-layer resilience (§4.3) mitigates link and backend fail-ures. We also outline datapath optimizations that make these mechanisms practical at scale (§4.4). 

## 4.1 Phase 1: Dynamic Orchestration 

The first phase of TENT’s execution pipeline is dynamic orchestration. Unlike Mooncake TE, which statically binds buffers to fixed backends (§2.1), TENT determines the trans-port for each submitTransfer at request time. Given the source and destination segment metadata, the orchestrator enumerates all reachable NICs and device fabrics (§3.1) and constructs the set of feasible transport options. It then intersects the capabilities of both endpoints and applies a tier-aware policy to select the highest-performance direct path available. When a fast fabric is present (such as NVLink for intra-node GPU-to-GPU transfers or tier-1 RDMA for GPUDirect), TENT binds the request to that backend, avoid-ing the communication silos created by static binding. When no direct GPU-direct path spans the endpoints (for example, on consumer GPUs without GPUDirect or deploy-ments where inter-GPU links are limited to MNNVL), TENT synthesizes a staged multi-hop route. ## 4.2 Phase 2: Telemetry-Driven Slice Spraying 

The second phase tackles the state-blind striping problem (§2.2). Given the transport plan produced by the previous phase, TENT decomposes large transfers into fine-grained slices and schedules each slice across the feasible NICs and device fabrics. Rather than hashing or round-robin placement, TENT schedules each slice to the rail expected to complete it fastest based on live telemetry. 

Slice Decomposition. For elephant flows, the payload is split into slices with a configurable minimum size (64 KB by default). The granularity is small enough that no single slice holds a rail for long, which helps reduce head-of-line blocking, yet large enough to amortize the enqueue and completion costs. For extremely large requests, we also cap the total number of slices to bound control-plane overhead. Each slice is issued as an independent work request that writes directly to its final destination offset, so slices from the same transfer may complete in any order without requiring CPU-side reordering.

Once the local NIC is selected, TENT maps it to a remote endpoint. To minimize switch hops and latency, the orchestra-tor employs a topology-aligned mapping strategy. At configurable intervals (e.g., every 30 s), the scheduler resets the learned parameters and accumulated penalties. This guarantees that previously degraded paths are periodically reintegrated into the resource pool once their performance recovers. 

## 4.3 Phase 3: Proactive Dual-Layer Resilience 

The third phase focuses on resilience under fabric churn (§2.3). Instead of treating failures as rare, control-plane events that require manual intervention, TENT embeds fault detec-tion and recovery directly into the data plane and operates 7at two layers: individual links (rails) and whole transport backends. At the link level, TENT monitors both implicit and explicit signals. The telemetry loop from Phase 2 naturally detects struggling rails as their predicted completion times grow rel-ative to peers; in addition, backends surface explicit errors such as RDMA completion failures, timeouts, or GPU-side events. When a rail crosses a degradation threshold, the sched-uler applies soft exclusion by effectively setting its cost to infinity and removing it from the candidate set, without in-voking heavyweight reconfiguration. A background prober periodically sends lightweight heartbeat slices to excluded rails; once a rail proves responsive and stable, its cost is reset and it is gradually re-admitted into the scheduling pool. Failures manifested as slice timeouts or hard errors are han-dled through idempotent per-slice re-execution. TENT writes each slice to an absolute destination offset, so retrying a slice is safe even if a previous attempt partially succeeded (for instance, data was written but the completion notification was lost). When a slice fails, the orchestrator retries it on an alter-native local–remote NIC pair drawn from the topology graph, prioritizing reliability over latency. Although these retries by-pass the predictive cost model, their resource consumption is still included in the global queue statistics used by Phase 2, so recovery traffic does not starve unrelated flows. At the transport level, entire backends may become unus-able, e.g., when a driver bug invalidates all NVLink paths on a node. Because Phase 1 already computed a candidate execution plan with multiple transports, TENT can perform backend substitution: when a backend reports a fatal error, the orchestrator promotes the next-best transport (e.g., falling back from NVLink to RDMA or TCP) for subsequent slices of the affected transfers. This transition happens entirely inside the runtime and requires no exception handling or reconfigura-tion at the application level. From the caller’s perspective, the transfer continues to completion with, at worst, a temporary performance dip. 

## 4.4 Low-Overhead Datapath 

These phases are only useful if the engine’s own overhead is negligible compared with the data it moves. TENT imple-ments a low-overhead datapath that keeps CPU usage minimal while sustaining high concurrency across large numbers of slices. When an application thread calls allocateBatch , the or-chestrator allocates a lightweight control block for the batch. Subsequent submitTransfer calls invoke the orchestration logic to select the transport and, when needed (e.g., RDMA), divide each logical transfer into slices. These slice descriptors are pushed into lock-free multi-producer, single-consumer ring buffers shared with worker threads, allowing application threads to return immediately without blocking on hardware availability. Pinned worker threads asynchronously dequeue slices and issue them to the underlying transport. To reduce submis-sion overhead, each worker opportunistically batches multi-ple slices into a linked list of work requests and posts them with a single transport call (such as a single ibv_post_send 

for many RDMA writes). Each worker owns its own queue pair or transport context, eliminating doorbell contention and stabilizing high-throughput submission. On the wire, TENT relies on one-sided operations that place data directly into the destination buffer at the slice’s absolute offset, enabling fully zero-copy, CPU-bypass reception even when slices arrive out of order. To efficiently track large numbers of outstanding slices, TENT uses hierarchical atomic counters. Worker threads poll completion queues in batches and, for each completed slice, atomically decrement a small counter in the corresponding batch’s control block. Applications observe only these coarse-grained counters (e.g., batch X has N remaining slices) rather than tracking per-slice state. This keeps the completion path simple, cache-efficient, and non-blocking, ensuring that the datapath remains light enough to support TENT’s higher-level orchestration, scheduling, and resilience mechanisms without becoming the bottleneck itself.

We examine whether TENT delivers its goals: improving the performance of latency-critical LLM workloads, remaining portable across heterogeneous fabrics/vendor stacks, and pro-viding practical resilience under realistic failures. 

Testbed and Baselines. All experiments run on GPU clus-ters configured to mirror current large-scale LLM serving de-ployments. Our primary testbed consists of nodes with eight NVIDIA H800 GPUs (80 GB HBM each), eight 200 Gbps RoCE NICs, and dual-socket Intel Xeon Platinum 8468V CPUs. Within a node, GPUs are connected by NVLink, while NICs attach via a hierarchical PCIe topology spanning NUMA domains. This combination of NVLink, PCIe, and multi-rail RDMA yields exactly the heterogeneous fabric that TENT is designed to orchestrate. Mooncake TE and UCCL-P2P are configured with their rec-ommended multi-rail setups (uniform striping or per-region NIC pinning), while NIXL uses its default UCX policy, which selects a small set of best NICs based on static transport prop-erties. 85.1 Performance

5.1.1 KVCache Reuse with SGLang HiCache 

To quantify the impact of TENT on user-visible latency and throughput for KVCache intensive LLM serving, we inte-grate TENT as a new backend for SGLang HiCache [27]. We study three configurations: a baseline with no HiCache (KVCache restricted to GPU memory), HiCache with Moon-cake TE, and HiCache with TENT. Both HiCache variants use a 600 GB KVCache budget and identical cache poli-cies; only the transfer engine differs. We report input-token throughput, average and P90 time-to-first-token (TTFT), and round-specific average TTFT for rounds 1, 5, and 10. Table 2 summarizes the results. Mooncake TE, in contrast, always routes GPU-to-GPU transfers through RDMA and uniformly stripes large transfers across NICs without considering topology or instantaneous rail health. TENT’s slice spraying converts NVLink into the preferred path for intra-node KV flows while still aggregating multi-rail RDMA bandwidth for inter-node traffic. The resulting fabric-aware data movement translates directly into lower TTFT and higher throughput for KVCache reusing workloads.

Device-to-Device. We next ask whether these benefits carry over to GPU-originated flows, where PCIe and NUMA topol-ogy introduce strong asymmetries across rails. Each H800 GPU in our testbed has one tier-1 NIC with the closest PCIe/N-VLink affinity, and three tier-2 NICs on the same NUMA node. For small blocks, the tier-1 NIC remains dominant; for large blocks, the tier-1 NIC becomes saturated, and TENT begins recruiting tier-2 NICs because the parallel bandwidth outweighs their access penalties. Per-NIC byte counters confirm this behavior: roughly half of the data is sent on the tier-1 NIC, with the remainder spread across tier-2 NICs. This is neither single-rail execution nor naive striping. Instead, TENT selectively adds rails only when doing so reduces the total service time, which matches the communication characteristics of KVCache and model-update transfers and explains the sharp throughput gains at larger block sizes.

## 5.3 Reliability 

We finally evaluate whether TENT’s dual-layer resilience turns common fabric failures into short-lived performance perturbations rather than application-visible errors. We draw on both production experience and targeted fault injection. 

Production Experience. TENT has been deployed on mul-tiple large-scale industrial LLM platforms. At one major AI service provider, TENT handles nearly all production infer-ence traffic on a thousand-GPU cluster and has been running continuously for over a year. During this period, operators observe frequent interconnect churn, including per-rail slow-downs, transient link flaps, and occasional device resets. De-spite this, TENT maintains stable throughput and tail latency without requiring any application-level failover logic, and without exposing transport errors to upstream systems. This experience supports the design choices of TENT: treat-ing transport health as a first-class signal, separating link-level and backend-level resilience, and centralizing error handling in the transfer engine are sufficient to keep production systems operating smoothly through routine fabric disruptions. 

Failure Injection in Testbed. To study failure behavior more systematically, we inject controlled NIC failures on the H800 testbed while continuously issuing 64 MB transfers. In one experiment, we force a failure on NIC 0 at t = 1000 ms and mark it as recovered at t = 3000 ms. Figure 10 shows the resulting throughput over time. When NIC 0 fails, in-flight transfers on that rail are aborted. TENT immediately marks the rail unavailable, reroutes sub-sequent slices to other NICs, and re-executes affected slices on healthy rails. This yields a brief but visible throughput dip lasting less than 50 ms during the transition; afterwards, the system continues in a slightly degraded state with reduced aggregate bandwidth. TENT is configured in this experiment to reset link status every second (longer intervals are used in production), and these periodic health checks produce small throughput fluctuations as TENT probes whether previously degraded rails have recovered. When NIC 0 is restored at 

t = 3000 ms, TENT reintegrates the rail within 26 ms and throughput quickly returns to its original level. At no point does the application observe a failed transfer: all retries, path substitutions, and health checks are handled inside TENT. Taken together, the production deployments and fault-injection experiments show that TENT provides practical reli-ability for large-scale LLM clusters: common NIC and link failures manifest as short-lived performance blips rather than persistent outages, and the system recovers high-performance routes automatically without application intervention. 

## 6 Related Works 

LLM Communication Frameworks. Collective communi-cation libraries like NCCL [8, 19], RCCL [2], HCCL [39], and others [7, 24, 28] extend group primitives to GPUs. While efficient for training, they assume homogeneous, static topolo-gies and are often vendor-coupled, limiting their use in dis-aggregated serving. In contrast, Point-to-point (P2P) frame-works such as Mooncake TE [25], NIXL [21], DLSlime [5], NVSHMEM [23], and UCCL-P2P [39] provide direct mem-ory access suited for dynamic workloads like prefill-decode disaggregation [13, 27, 31, 37], KVCache storage [4, 16], and RLHF [29]. However, as mentioned in §2, existing P2P en-gines typically rely on static transport binding (e.g., fixed RDMA or NVLink paths) and lack runtime adaptation. Sep-arately, Expert Parallelism (EP) frameworks [15, 39] couple communication with computation to optimize MoE routing, prioritizing kernel-level efficiency over the unified abstraction of diverse heterogeneous interconnects. 

Transport Orchestration. UCX [30] offers unified memory-to-memory abstractions over RDMA, TCP, and SHM, select-ing paths using established priorities at connection setup. Sys-tems such as NIXL [21] and DLSlime [5] unify remote mem-ory and storage but generally rely on application-directed backend choices. In contrast, TENT adopts a declarative inter-face that decouples intent from execution, enabling dynamic per-request path selection across heterogeneous fabrics, which is beneficial for disaggregated LLM serving where link con-ditions evolve over time. 

Transport-Level Performance Optimization. While P2P engines (e.g., Mooncake TE [25], NIXL with UCX [21, 30]) support multi-rail RDMA, they often rely on static strategies like round-robin. TENT improves this via telemetry-driven slice scheduling (§4.2). Prior works [6, 11, 26, 33] have opti-mized low-level RDMA efficiency (e.g., doorbell batching). Research also identifies performance bottlenecks in multi-tenant NICs [12, 32] and addresses scale-out QP scalability 12 challenges via techniques like eRPC [10], Flock [17], and SMART [26]. TENT leverages these low-level insights within a higher-level declarative runtime. 

Fault Tolerance. Most existing frameworks rely on reactive fault handling. UCX [30] uses callbacks to report failures, requiring the application to explicitly close and recreate in-valid endpoints. NCCL [19] supports asynchronous detection (AER) but halts ongoing operations, requiring global com-municator reinitialization. NIXL [21] lacks explicit recovery mechanisms, deferring error handling to underlying transports. In contrast, TENT integrates dual-layer resilience, enabling proactive link-level failover and transparent transport-level recovery without user intervention. 

## 7 Conclusion 

TENT overcomes the operational rigidity of modern GPU clusters by shifting data movement from imperative binding to declarative slice spraying. By unifying heterogeneous in-terconnects into a dynamic, self-healing resource pool, TENT transparently optimizes multi-rail bandwidth and masks hard-ware churn. Validated at production scale, it delivers up to 1.36 × higher serving throughput and 26 .1% faster model up-dates, demonstrating that a resilient, declarative data plane is essential for next-generation disaggregated AI infrastructure.

## 摘要
网络流量监控对于大多数网络任务至关重要，例如识别和阻断攻击、定位故障，以及对重流量进行流量工程和重路由以维持高吞吐量。流量监控中的一个重要指标是寻找 Top-K 重流，即流量中最大的 K 个流。可编程网络允许直接在数据平面执行高级网络分析。近年来，已经提出了多种在单个交换机内高效寻找 Top-K 重流的解决方案。然而，有时我们需要寻找全局 Top-K 流。现有的全局 Top-K 检测方案使用集中式控制器来收集和聚合每个交换机中执行的测量结果。然而，将信息发送到控制平面，再由控制器将信息发回交换机的过程可能非常耗时。为了能够检测并缓解短时事件，需要完全在数据平面内运行的解决方案。在本文中，我们提出了 NODE，一种完全在数据平面运行的全网 Top-K 检测算法。NODE 允许交换机聚合来自网络中所有其他交换机的信息，并确保最终所有交换机都持有一个相同的全局 Top-K 表。我们证明了 NODE 在合成和真实数据集上都能成功检测全局 Top-K 流，在每个交换机使用不到 300KB 内存的情况下，召回率超过 95%。

## Abstract
Monitoring network traffic is crucial for most network tasks, such as, identifying and blocking attacks, pinpointing failures and engineering and rerouting heavy traffic to maintain high throughput. One important metric when monitoring the traffic is finding the top-k heavy flows, that is the k heaviest flows in the traffic. Programmable networks allow performing advanced network analysis right in the data plane. In recent years, various solutions have been proposed for efficiently finding the top-k heavy flows within a single switch. However, at times we may need to find the global top-k flows. Existing solutions for global top-k detection use a centralized controller that collects and aggregates the measurements performed in each of the switches. Yet, the process of sending information to the control plane and then having the controller send back the information to the switches can be very lengthy. In order to be able to detect and mitigate short-lived events, solutions that work completely within the data plane are needed. In this paper we present NODE, a network-wide top-k detection algorithm that operates exclusively in the data plane. NODE allows the switches to aggregate information from all other switches in the network, and ensures that eventually all switches hold an identical global top-k table. We show that NODE manages to detect global top-k flows on both synthetic and real traces, with a recall rate of over 95\% while using less than 300KB per switch.

*   **研究背景**：
    *   **现有方案局限性**：目前的单交换机 Top-K 检测已较成熟，但全网检测通常依赖**中心化控制器**。控制器方案存在高延迟（无法捕捉短时突发流量）和采样精度低（为减少开销而牺牲准确性）的问题。
    *   **分布式挑战**：在纯数据平面实现全网聚合面临严峻挑战，包括可编程交换机极度受限的内存和计算资源、内存访问顺序限制（PISA 架构），以及重流可能被分散在多个交换机中导致局部检测失效（Split Flows）的问题。

### 2. 方法论
NODE 的核心思想是将检测过程分为三个确定性的阶段，并利用 **Swish** 框架在交换机间共享状态，无需控制器干预。

*   **核心流程**：
    1.  **本地 Top-K 创建**：每个交换机独立运行本地算法（如 Precision）识别本地重流，并存入 `L-TopK` 表。
    2.  **全局计数聚合（Aggregation Round）**：交换机将本地 Top-K 信息广播。接收方检查收到的流 ID 是否存在于自己的本地表中，若匹配则累加计数。此阶段结束后，每个交换机持有其本地重流的**全局准确计数**。
    3.  **全网表合并（Consolidation Round）**：交换机再次广播带有全局计数的流信息。接收方通过比较计数大小，动态替换 `G-TopK` 表中的条目。
*   **关键技术细节**：
    *   **内存访问优化**：针对 PISA 架构，NODE 在聚合阶段采用“ID 优先”布局（先匹配 ID 再更新计数），在合并阶段采用“计数优先”布局（先比较计数再决定是否替换 ID），巧妙绕过了硬件流水线的限制。
    *   **一致性保证**：论文通过数学证明，确保在合并阶段结束后，全网所有交换机最终维护的 `G-TopK` 表是完全一致的。
    *   **数据结构**：使用了 Snapshot（快照）、Sum（求和）、G-TopK（全局表）和 Query（查询）等五种表结构。

### 6. 主要结论与发现
*   **高准确性**：在各种数据集下，NODE 的 Top-K 召回率均超过 **95%**，表现接近无资源限制的控制器方案。
*   **低内存占用**：仅需不到 300KB 内存即可处理千万量级的流，远低于 FlowRadar 等方案（后者通常需要数 MB 甚至 20MB 以上）。
*   **一致性**：成功实现了全网交换机对全局 Top-K 视图的同步。
*   **通信权衡**：虽然原生 NODE 的消息量随交换机数量平方增长，但通过**聚类方法**可以将通信开销降低一个数量级，且召回率损失较小。

### 7. 优点
*   **纯数据平面运行**：彻底摆脱了控制器的延迟瓶颈，能够实时响应网络中的短时突发事件。
*   **硬件友好**：针对 P4 交换机的流水线限制进行了专门设计，不使用复杂的循环或多次重循环（Recirculation）。
*   **内存效率极高**：利用流 ID 共享和确定的聚合逻辑，极大地压缩了状态存储空间。
*   **数学证明**：提供了全网一致性的理论保证，增加了算法的可靠性。

### 8. 不足与局限
*   **通信开销**：在大型网络中，全网广播会导致显著的控制流量。虽然提出了聚类方案，但在超大规模网络中的扩展性仍面临挑战。
*   **配置复杂性**：依赖于 Swish 框架，且需要占用交换机较多的流水线阶段（12 个 Stage），可能会挤占其他网络功能（如复杂路由或防火墙）的硬件资源。
*   **动态性处理**：论文主要关注一个测量周期内的 Top-K，对于极高动态变化下的流连续追踪能力尚待进一步验证。

## 摘要
本文介绍了 NEST（网络强制会话类型），这是一种将应用层协议监控移至网络架构中的运行时验证框架。与以往对应用程序代码进行插桩或封装的工作不同，我们合成了直接在数据平面中强制执行协议的数据包级监控器。我们开发了从会话类型生成网络级监控器的算法，并将其扩展以处理丢包和重排序问题。我们在 P4 中实现了 NEST，并在包括微服务和网络功能模型在内的应用上进行了评估，结果表明网络级监控器能够强制执行现实且复杂的协议。

## Abstract
This paper introduces NEST (Network-Enforced Session Types), a runtime verification framework that moves application-level protocol monitoring into the network fabric. Unlike prior work that instruments or wraps application code, we synthesize packet-level monitors that enforce protocols directly in the data plane. We develop algorithms to generate network-level monitors from session types and extend them to handle packet loss and reordering. We implement NEST in P4 and evaluate it on applications including microservice and network-function models, showing that network-level monitors can enforce realistic non-trivial protocols.

#### 1. 核心问题与整体含义（研究动机和背景）
论文的核心问题是如何在**网络数据平面（Data Plane）**中直接强制执行应用层通信协议。
*   **背景**：会话类型（Session Types）是验证分布式系统消息传递协议的成熟理论，但传统监控多在应用层（如 Rust/Go 库或代理）实现。
*   **动机**：应用层监控容易被受损的主机绕过，且在多租户云环境中缺乏信任保障。将监控下沉到网络交换机或网卡（NIC），可以实现“默认关闭”的网络安全架构，在非法数据包进入网络深处前将其丢弃，节省资源并增强安全性。

#### 2. 方法论
NEST 提供了一套从高层协议描述到底层网络配置的自动合成框架：
*   **核心思想**：将局部会话类型（Local Session Types）转换为有限状态机，并将其映射为 P4 编程语言中的**匹配-动作表（Match-Action Tables, MATs）**。
*   **关键技术细节**：
    *   **监控器合成算法**：开发了专门的算法，将分支、循环等会话类型结构转换为 P4 状态寄存器和流表项。
    *   **网络异常处理**：引入了**会话序列号（SSN）**机制，使监控器能够识别并处理 TCP 层的丢包重传、重复包和乱序包，而无需在交换机上进行大规模缓存。
    *   **共享入口支持**：支持单个网络设备同时监控多个终端节点和多种协议角色。
    *   **端侧 API 生成**：自动生成 Python API，确保终端应用程序发出的数据包包含监控所需的特殊会话头部（Session Header）。
*   **形式化保证**：论文证明了监控器的**完备性（Soundness）**，即只要终端节点遵循协议，监控器绝不会误杀合法流量。

#### 5. 实验数量与充分性
*   **实验规模**：针对 10 种不同复杂度的协议进行了测试，每种协议均覆盖了 UDP、可靠 TCP 和不可靠 TCP 三种网络环境。
*   **充分性**：实验涵盖了从简单的线性协议到复杂的带有嵌套循环和多方分支的协议。通过注入网络损伤（丢包、乱序）来验证 TCP 监控逻辑的鲁棒性，实验设计较为全面且客观。

#### 6. 主要结论与发现
*   **有效性**：NEST 能够准确拦截所有违反协议的数据包，且在 TCP 模式下，一旦检测到违规，能立即关闭相关会话。
*   **透明性**：对于遵循协议的节点，NEST 监控器表现透明，不会引入误报或干扰正常的重传机制。
*   **可行性**：证明了复杂的应用层逻辑可以被压缩进资源受限的 P4 数据平面匹配表中。

#### 7. 优点
*   **架构创新**：首次将会话类型理论与可编程数据平面（P4）结合，实现了硬件级的协议强制执行。
*   **理论与实践结合**：不仅提供了形式化证明，还解决了实际网络中的丢包和乱序等工程难题。
*   **自动化程度高**：从 Scala DSL 到 P4 代码及端侧 API 的全自动生成，降低了部署门槛。

#### 8. 不足与局限
*   **硬件限制**：受限于 P4 交换机的内存和寄存器资源，无法支持状态空间无限大的协议（如某些非半双工的递归协议）。
*   **分片处理**：当前版本假设一个应用消息对应一个数据包，尚不支持跨包的消息分片（Fragmentation）。
*   **协议约束**：要求协议必须是**半双工（Half-duplex）**且**输出活跃（Output-live）**的，限制了某些极高性能并发场景的应用。
*   **加密挑战**：如果应用层数据完全加密且不暴露头部信息，网络层监控器将难以解析消息标签（Label）。

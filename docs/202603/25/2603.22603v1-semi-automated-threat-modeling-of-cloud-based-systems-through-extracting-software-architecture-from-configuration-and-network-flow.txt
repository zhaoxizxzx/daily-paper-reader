# Semi-Automated Threat Modeling of Cloud-Based Systems Through Extracting Software Architecture from Configuration and Network Flow 

Nicholas Pecka, Member, IEEE, Lotfi Ben Othmane, Bharat Bhargava, and Renée Bryce Fellow, IEEE, 

Abstract —Traditional threat modeling occurs during design, but cloud deployments introduce unanticipated threats—especially multi-stage attacks chaining vulnerabilities across trust boundaries. Existing security tools analyze components in isolation, cannot detect architectural threats from system composition, and cannot validate runtime behavior against configured policies. This gap leaves organizations vulnerable to attacks exploiting architectural weaknesses. This paper addresses this gap through a key innovation: automatically inferring system architecture from runtime observations to enable continuous threat modeling. Our methodology combines static configuration analysis with observed network flows to construct architecture graphs reflecting actual operational behavior, then applies systematic threat detection using platform-agnostic abstractions (components, domains, interfaces, access policies, flows). This enables consistent threat identification across bare metal, Kubernetes, and cloud infrastructure without manual diagram maintenance. We validate the methodology using a supply-chain system with ML components deployed on all three platforms, injecting 17 infrastructure and ML threats. Results show detection of all 17 threat types across all platforms, while existing security tools detected only 6-47% with zero ML threat coverage, confirming the necessity of runtime-aware, architecture-level threat analysis. 

Index Terms —Architecture recovery, configuration manage-ment, Kubernetes, threat modeling, runtime security analysis, dynamic analysis, cloud security.

Static Security Analysis. Vulnerability scanners (Trivy [14], Checkov [15], KubeLinter [16]) and CSPM tools (Prisma Cloud [17], Wiz [18], AWS Security Hub [19]) detect known CVEs [20] and misconfigurations through static analysis of container images, IaC templates, and cloud configurations. However, they analyze components in isolation, are platform-specific, and cannot detect architectural threats emerging from system composition or runtime behavior contradicting con-figured policies. Our methodology provides platform-agnostic architectural threat identification through runtime analysis. 

Runtime Monitoring and Network Security. Runtime mon-itoring tools (Falco [21], AWS GuardDuty) and service mesh platforms (Istio [22], Linkerd [23]) provide security teleme-try through system call observation, anomaly detection, and traffic visibility. While valuable for incident detection and microsegmentation, they produce event-level alerts or ob-servability dashboards without classifying architectural threats using frameworks like STRIDE [24] or constructing attack graphs. Our approach correlates runtime observations with configuration and trust boundaries to generate structured threat models. 

Automated Threat Modeling. Recent research automates threat modeling through LLMs and runtime adaptation. Aus-pex [25] uses LLMs to generate STRIDE threats from dia-grams but requires accurate up-to-date architectural diagrams. Our methodology infers architecture from running systems, addressing diagram drift [10]. ACTISM [26] and ThreMo-LIA [27] target domain-specific architectures (automotive, LLM applications) while our approach generalizes across deployment paradigms. 

Constructing Threat Modeling Rules. Fu et al. [28] propose an approach that analyzes vulnerabilities disclosed in the NVD dataset, classifies them into a threat taxonomy, and derives threat-modeling rules with high accuracy. These rules enable identification of threats applicable to a target system. By grounding threat identification in continuously updated vulnerability disclosures, the approach helps keep pace with an evolving threat landscape and supports detecting newly emerging threats in existing systems. 

Our Contributions. We address three gaps: (1) architecture inference from runtime observations without requiring pre-existing diagrams [29], [30], (2) platform-agnostic abstractions applying uniformly across bare-metal, Kubernetes, and cloud, and (3) ML/AI-aware threat modeling with built-in ML com-ponent recognition, achieving 100% threat detection versus 0% ML threat coverage by existing tools. B. Semi-automated Threat Modeling Process 

Our semi-automated threat modeling methodology consists of five iterative stages (Figure 1): (1) Data Collection ex-tracts configuration data and runtime flows from the deployed system, (2) Architecture Construction builds an architecture graph representing the actual operational state of the system, capturing both the intended design and emergent runtime behavior, (3) Threat Modeling Mapping maps the inferred architecture to threat modeling constructs compatible with established frameworks (PyTM, Microsoft Threat Modeling Tool), (4) Threat Identification applies systematic pattern-based detection algorithms to identify security weaknesses, and (5) Mitigation Planning prioritizes identified threats by risk score and generates platform-specific mitigation rec-ommendations. The methodology operates continuously, re-analyzing the system as configurations change and new flows emerge, maintaining an up-to-date threat model that reflects the evolving security posture of the running system. The core of our approach lies in two main processes: (1) in-ferring the system architecture from runtime observations, and (2) identifying and analyzing architectural threats. Algorithm 1 describes how we extract platform-specific configuration data, correlate it with observed network flows to build an archi-tecture graph, and map this graph to platform-agnostic threat modeling constructs. Algorithm 2 details how we apply threat frameworks to identify security weaknesses, correlate them with known vulnerabilities, and generate actionable mitigation recommendations. 

1) Architecture Inference from Runtime Observations: We transform running systems into structured threat models by collecting system configuration S = ( D, C, I, A) (domains, components, interfaces, policies) and runtime flows H via 2platform-specific monitoring tools. The architecture inference process constructs graph G = (V, Eg , Z) where vertices represent components, edges capture observed flows, and zones partition security domains. We correlate network-level identifiers (IPs, service names) to logical components using platform-specific strategies, then validate edges against poli-cies to identify anomalous flows. Finally, we map platform-specific graphs to threat modeling constructs: zones become trust boundaries B, components become threat elements Et

(processes/datastores/entities), and flows become dataflows F

enriched with security metadata (Table II). 

2) Threat Identification and Mitigation Planning: Once we have constructed the threat model T = ( B, Et, F) through architecture inference (Algorithm 1), the second phase applies security frameworks to identify threats, correlates findings with known vulnerabilities, and generates prioritized miti-gation recommendations. This process enriches the abstract threat model with actionable security intelligence. We begin by applying threat frameworks (such as STRIDE [24], MITRE ATT&CK [35], and OWASP ML Secu-rity Top 10 [36]) to each threat model element and dataflow, generating a set of potential threats Θ based on element types and communication patterns. Elements classified as processes face threats like elevation of privilege and tampering, while datastores are vulnerable to information disclosure and denial of service. Dataflows that cross trust boundaries receive elevated scrutiny as they represent transitions between security domains. The methodology then correlates architectural threats with known vulnerabilities by querying vulnerability databases K

using component version information. This integration com-bines architectural analysis with component-level security in-telligence, enabling detection of threats that emerge from both deployment topology and software vulnerabilities. When threat intelligence feeds L are available, we further enrich the threat set with contextual information about active attack campaigns and emerging threat patterns. Risk prioritization considers both intrinsic threat severity and contextual factors Ψ such as external exposure, data sensitivity, and compliance requirements. The methodology calculates risk scores for each identified threat and applies organizational security policies P to filter and prioritize find-ings. Finally, the methodology generates platform-specific mitiga-tion recommendations R by querying a mitigation knowledge base that maps abstract threats to concrete security controls. Where possible, we generate automated remediation configu-rations Am that can be directly applied to the target platform (e.g., NetworkPolicy YAML for Kubernetes, Security Group rules for AWS). C. Illustrative Example: Platform-Agnostic Threat Detection 

To illustrate how our platform-agnostic methodology op-erates across diverse infrastructures, Figure 2 shows the de-tection process for T01 (Unauthenticated External Access). The same abstract threat detection logic applies uniformly across platforms by mapping platform-specific configurations to our abstract constructs. For example, detecting T01 requires identifying: (1) an external-facing interface ( I), (2) absence of authentication policy ( A), and (3) observed external traffic (H). On Kubernetes, this manifests as a LoadBalancer Ser-vice without authentication annotation and a NetworkPolicy gap. On bare-metal, it appears as a 0.0.0.0 binding with missing iptables rules. On AWS, it emerges as a Security Group permitting 0.0.0.0/0 ingress without IAM authentication requirements. Despite differing implementations, the abstract detection algorithm remains identical, validating our method-ology’s platform independence. IV. E VALUATION 

To validate that our methodology automatically infers archi-tecture from runtime observations and detects threats across heterogeneous deployments, we deployed a software supply chain system across three platforms and evaluated architecture inference and threat identification capabilities using platform-agnostic abstractions. The supply chain consists of eight components (Gitea, Jenkins, Docker Registry, Application, PostgreSQL, MLflow Model Registry, ML Inference Ser-vice, Training Data Storage) representing a modern DevOps workflow with integrated MLOps. The system was deployed across bare-metal infrastructure, a Kubernetes cluster [37], and Amazon Web Services [12]. We intentionally introduced sev-enteen types of architectural threats into the supply chains (ten traditional infrastructure threats T01-T10 plus seven AI/ML-specific threats T11-T17) using platform-specific mechanisms but with conceptually identical security implications. We then applied our five-stage semi-automated threat modeling methodology to each deployment to validate cross-platform generalization.

Framework Alignment. Traditional infrastructure threats (T01-T10) align with STRIDE categories (Spoofing, Tam-pering, Information Disclosure, Denial of Service, Elevation of Privilege, Repudiation) and MITRE ATT&CK techniques (T1190, T1078, T1040, T1021, T1525, T1199, T1562.002, T1552, T1496). ML/AI-specific threats (T11-T17) map to OWASP ML Security Top 10 [36], OWASP LLM Top 10 [42], MITRE ATLAS [41] (AML.T0040, AML.T0020, AML.T0024, AML.T0010), and NIST AI RMF [43] require-ments. This systematic alignment ensures our threat taxonomy represents recognized security concerns rather than ad-hoc selections. 

Note on Threat Types vs. Instances : We distinguish between threat types and threat instances . A threat type (T01-T17) represents a category of architectural vulnerability, while an instance represents a specific occurrence of that threat type in the deployed system. For example, T08 (Exposed Secrets) is a single threat type that manifests as multiple instances when plaintext credentials appear in different configuration files (gitea-secrets, database-credentials, application environment variables, etc.). Our evaluation reports both the number of threat types detected and the total number of threat instances identified across all components.

V. R ESULTS AND ANALYSIS 

We present validation results demonstrating that our methodology successfully performs platform-agnostic threat modeling across diverse deployment paradigms. 

A. Cross-Platform Methodology Validation 

Our methodology successfully validated all core capabili-ties across three deployment platforms. The methodology in-ferred architecture graphs from runtime observations, mapped platform-specific mechanisms to abstract constructs, and iden-tified all 17 intentionally introduced threat types (10 traditional infrastructure threats T01-T10 plus 7 ML-specific threats T11-T17) on bare-metal, Kubernetes, and AWS deployments using identical abstract detection logic. ML threat detection on bare-metal identified 49 total threat instances (42 traditional + 7 ML-specific), Kubernetes detected 47 instances (40 traditional + 7 ML-specific), and AWS detected all 17 threat types (T01-T17), demonstrating consistent cross-platform detection with the same abstract threat logic applied through platform-specific mechanisms. In comparison, existing security scanning tools detected only 6-47% of all threats with default configurations (Ta-ble IV). Static IaC scanners with default configurations de-tected 6-18%, while CSPM tools with default configurations detected 41-47%. Critically, no existing tool detected any of the seven ML-specific threats (0/7) without hand-crafted custom rules. Furthermore, none can detect the three runtime-confirmed ML threats (T12, T14, T15) which require ob-serving ML component communication patterns. This demon-strates that static configuration analysis alone is insufficient for comprehensive threat detection in modern distributed systems, particularly for emerging ML/AI deployment architectures. Architectural threats that emerge from component composi-tion, trust boundary crossings, runtime behavior, and ML-specific access patterns require the semi-automated correlation capabilities and built-in ML component recognition provided by our methodology.

Algorithm Complexity and Scalability. The architecture graph construction (Algorithm 1) has O(n²) worst-case com-plexity for flow correlation, though realistic systems exhibit sparse communication patterns reducing practical complexity. Threat detection (Algorithm 2) scales linearly O(n) with the number of components and dataflows. Our evaluation ana-lyzed small-scale deployments: 3-5 infrastructure units per platform (VMs for bare-metal, pod groups for Kubernetes, EC2 instances for AWS) hosting eight application components distributed across three trust zones. For large-scale deployments (50+ services), flow correla-tion complexity depends on communication density—sparse service meshes maintain near-linear scaling while densely connected systems approach quadratic complexity. Threat de-tection remains linear regardless of scale. Memory requires approximately 50-100 MB per 100 components. Practical strategies such as zone-based analysis and incremental updates can mitigate scaling challenges. Production-scale validation remains future work.

VI. T HREATS TO VALIDITY 

Internal Validity. Communication flows must occur during observation; we mitigated this by executing complete end-to-end workflows before analysis. Flow correlation uses connec-tion metadata (IP, port) rather than packet contents, enabling encrypted traffic handling via platform-native tools (Cilium Hubble, VPC Flow Logs). 

Methodology Scope. Our approach detects architectural 

threats from deployment topology. It cannot detect application-level vulnerabilities, implementation bugs, or supply chain artifact attacks—these require complementary techniques. 

External Validity. Evaluation focused on supply chain archi-tecture (8 components, 3 trust zones) [47], [48]. Validation on additional system types (microservice meshes, serverless) remains future work [49], [50]. Our 17-threat taxonomy rep-resents common vulnerabilities [40] but is not exhaustive. 

Construct/Reliability. Platform-specific threat implementa-tions required judgment on conceptual equivalence. Repro-ducibility is supported by open-source implementation [29]. VII. C ONCLUSION 

Traditional threat modeling during design becomes outdated as cloud deployments evolve, leaving organizations vulnerable to architectural threats from runtime behavior and system composition. Existing security tools analyze components in isolation, lacking the architectural context to detect cross-boundary attacks, privilege escalation paths, and ML-specific deployment vulnerabilities. This paper addresses this gap through a key innovation: automatically inferring system archi-tecture from runtime observations to enable continuous threat modeling without manual diagram maintenance. Our method-ology combines static configuration analysis with observed network flows to construct architecture graphs reflecting actual operational behavior, then applies systematic threat detection using platform-agnostic abstractions working uniformly across bare-metal, Kubernetes, and AWS. We validated this approach on a software supply chain system with ML components, injecting 17 infrastructure and ML threats. Results demonstrate detection of all 17 threat types across all platforms, while existing tools detected only 6-47% with zero ML coverage, confirming static scanners cannot validate runtime behavior and CSPM tools lack archi-tectural context for cross-component correlation. This is crit-ical because ML-enabled cloud systems face evolving attack surfaces where multi-stage attacks chain architectural weak-nesses—exposed trust boundaries, unauthorized cross-zone access, ML infrastructure misconfigurations—to compromise critical assets. Our methodology enables maintaining accurate threat models reflecting operational security reality, supporting continuous architecture-level assessment across heterogeneous deployments. Future work includes extending the methodology to addi-tional system architectures (microservice meshes, event-driven systems, IoT deployments) and incorporating automated re-mediation generation that produces platform-specific security configurations from identified threats.

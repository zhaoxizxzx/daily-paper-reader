## Abstract 

Investigating cybersecurity incidents requires collecting and an-alyzing evidence from multiple log sources, including intrusion detection alerts, network traffic records, and authentication events. This process is labor-intensive: analysts must sift through large volumes of data to identify relevant indicators and piece together what happened. We present a RAG-based system that performs security incident analysis through targeted query-based filtering and LLM semantic reasoning. The system uses a query library with associated MITRE ATT&CK techniques to extract indicators from raw logs, then retrieves relevant context to answer foren-sic questions and reconstruct attack sequences. We evaluate the system with five LLM providers on malware traffic incidents and multi-stage Active Directory attacks. We find that LLM models have different performance and tradeoffs, with Claude Sonnet 4 and DeepSeek V3 achieving 100% recall across all four malware sce-narios, while DeepSeek costs 15 × less ($0.008 vs. $0.12 per analysis). Attack step detection on Active Directory scenarios reaches 100% precision and 82% recall. Ablation studies confirm that a RAG archi-tecture is essential: LLM baselines without RAG-enhanced context correctly identify victim hosts but miss all attack infrastructure including malicious domains and command-and-control servers. These results demonstrate that combining targeted query-based filtering with RAG-based retrieval enables accurate, cost-effective security analysis within LLM context limits.

## 1 Introduction 

Security Operations Centers operate under increasing pressure: cyber threats grow in both volume and complexity, yet the analyst workforce has not kept pace. Modern Security Information and Event Management (SIEM) deployments routinely generate thou-sands of alerts per day, but empirical studies show analysts can only examine a small subset, resulting in fatigue and overlooked incidents [ 1–3 ]. Investigating a security incident (identifying com-promised assets, tracing attacker actions, understanding intrusion mechanisms) demands synthesis of evidence spread across intru-sion detection systems, network telemetry, and endpoint logs. This labor-intensive work strains limited analyst capacity, motivating research into intelligent systems that can accelerate incident analy-sis. Current detection mechanisms depend heavily on signatures and static rules, which cannot generalize to previously unseen attack patterns. Sophisticated adversaries exploit this gap through tech-niques like misconfigured certificate template abuse (e.g., ESC1 [ 4]), Kerberos ticket forgery [ 5 ], and stepwise lateral movement through network trust relationships. Such attacks can progress over hours or days, depositing forensic artifacts across disparate log sources (Suricata alerts, Zeek connection records, Windows authentica-tion events) where no individual entry reveals the complete attack chain. Recent machine learning approaches address alert triage and anomaly scoring [ 6, 7 ], but these methods yield classifications or rankings, not the narrative reconstruction that incident responders require. Large language models are well-positioned to address this prob-lem: they can interpret unstructured text, integrate scattered facts into coherent narratives, and communicate findings in natural lan-guage [ 8]. Feeding raw security logs directly into an LLM, however, is impractical. A single incident may produce millions of log entries, vastly exceeding model context limits [ 9 ]. These logs mix genuine attack indicators with overwhelming volumes of routine activity (normal authentications, benign connections, spurious alerts), bury-ing the attack signal. Additionally, incident response often demands rapid turnaround, while LLM inference remains comparatively slow. These challenges motivate the design of new AI-enabled security systems that extract relevant events from such large volumes of logs and generate incident reports interpretable by security analysts. We present a RAG-based system for security incident analy-sis that extracts relevant threat indicators by running targeted queries on security logs, mapping the attack indicators to the MITRE ATT&CK framework, and feeding the subsequent results to a LLM with context augmented by semantically similar events to gen-erate interpretable reports of security incidents. The framework operates in two modes: forensic question-answering and attack reconstruction. In the first mode, analysts pose specific questions (identifying infected hosts, command-and-control (C2) servers, or initial compromise vectors) and receive evidence-backed reports with explainable answers. In the second mode, the system recon-structs attack sequences by associating distinct forensic events into causal narratives of attack progression over time. Central to our approach is a security-aware context construction pipeline. Rather

than pass raw logs to the LLM, we extract through targeted queries Indicators Of Compromise (IOCs), compute aggregates of log data, and organize results into semantically meaningful chunks that pre-serve cross-event relationships. The RAG architecture is essential in extracting the most relevant information from multiple security logs for inclusion in the LLM context, enabling the model to gener-ate interpretable incident reports for security analysts. Our general architecture supports multiple LLM providers—both cloud APIs and locally deployed models—enabling organizations to balance performance, cost, and data sensitivity. We make the following contributions: 

Problem formulation for LLM-based incident analysis. We frame security incident analysis as two complementary tasks: (a) structured question-answering with type-specific matching func-tions that handle the semantics of security answers (IP addresses, hostnames, domain sets), and (b) attack sequence reconstruction that correlates discrete forensic events into causal narratives of attack progression. 

Security-aware RAG architecture with cross-event seman-tic reasoning. We design a retrieval pipeline tailored to security data: targeted queries extract indicators, aggregators compress thou-sands of events into patterns, and semantic chunking preserves cross-event relationships. This enables LLMs to identify attacks requiring correlation across discrete events, such as the detection of Active Directory certificate abuse by recognizing Subject/SAN mismatches across certificate request and issuance operations. This demonstrates contextual interpretation that distinguishes coordi-nated exploitation from individually legitimate activities. 

LLM provider evaluation with cost-effectiveness analy-sis. We evaluate eight LLM configurations spanning cloud APIs (Claude Sonnet 4, DeepSeek V3, GPT-4, GPT-4o, GPT-5-mini, GPT-5.2) and local deployment (Llama 3.1:70b, Cisco Foundation-Sec-8B) on malware traffic and Active Directory attack scenarios. The best-performing configurations (Claude Sonnet 4, DeepSeek V3) achieve 100% recall across all four malware scenarios that we analyzed, while attack step detection on Active Directory scenarios reaches 100% precision and 82% recall. DeepSeek achieves accuracy equiv-alent to Claude at 15 × lower cost ($0.008 vs. $0.12 per analysis), making it the most cost-effective cloud provider. Local deployment (Llama) provides data privacy with acceptable accuracy (95% re-call) for environments with strict data governance requirements. Notably, the security-specialized Cisco Foundation-Sec-8B model does not outperform general-purpose models at 90% recall, suggest-ing that reasoning capability matters more than domain-specific pretraining for complex forensic tasks. The system completes mal-ware analysis in 1–2 minutes per incident with cloud providers, while Active Directory attack analysis runs in under 1 minute per 5-minute window with local deployment, demonstrating real-time feasibility for both scenario families. To support reproducibility, we will release our implementation, including the query library, RAG pipeline, and evaluation scripts, along with detailed results for the malware traffic scenarios. The malware traffic datasets are publicly available [ 10 ]; the Active Di-rectory data originates from a private red team exercise and has not been publicly released. 

## 2 Related Work 

LLMs for Cybersecurity. Large language models have emerged as tools for cybersecurity automation, with recent surveys cataloging applications across vulnerability detection, malware analysis, threat intelligence, and incident response [ 8 ]. This body of work demon-strates both the potential and limitations of LLMs for security tasks. Early efforts adapted pre-trained models to cybersecurity text. SecureBERT [ 11 ] introduced a domain-specific BERT model trained on cybersecurity corpora, demonstrating improved performance on tasks involving MITRE ATT&CK terminology. CySecBERT [ 12 ]adopted domain adaptive pre-training to BERT on a large scale cybersecurity corpus to improve performance on cybersecurity NLP tasks such as Named Entity Recognition and Cyber Threat Intelligence. For threat taxonomy mapping, TRAM fine-tunes SciB-ERT to map reports to MITRE ATT&CK techniques [ 13 ]. These models provide strong semantic representations for classification and embedding tasks. Log-based anomaly detection represents a key application area. LogLLM [ 14 ] combines BERT for semantic extraction with Llama for sequence classification on system log datasets. LogPrompt [ 15 ]explores zero-shot anomaly detection through prompt engineering. These approaches focus on system logs (e.g., HDFS, Linux) and binary anomaly labels rather than security-specific logs with multi-stage attack patterns. Recent work explores Multi-agent LLM systems to cyber security tasks such as insider threat detection (Audit-LLM [ 6] demonstrates reduced false positives for insider threat detection) while SOC-specific systems like CORTEX [ 7] propose specialized agents for alert triage. Work on CTI automation [ 16 ] develops agents that ex-tract Indicators of Compromise from threat reports. These systems advance alert-level processing but focus on triage and summa-rization rather than answering specific questions about detected incidents. Examining operational use, Singh et al. analyzed queries from 45 SOC analysts, finding that LLMs function as cognitive aids that augment rather than replace expertise [1]. In contrast to prior work focusing on alert triage or classifica-tion, our work addresses end-to-end incident analysis formulated as question-answering over heterogeneous security logs, process-ing data from intrusion detection systems, network monitors, and authentication events. 

Retrieval-Augmented Generation. Retrieval-Augmented Gen-eration (RAG) addresses limitations of LLMs—hallucination, out-dated knowledge, and opacity—by conditioning generation on exter-nal evidence retrieved at inference time [ 17 ]. RAG has become the dominant paradigm for knowledge-intensive tasks where accuracy and traceability are required. Surveys categorize RAG architectures into retriever-centric, generator-centric, and hybrid designs, with recent advances in adaptive retrieval, iterative refinement, and self-reflection [18, 19]. Recent work applies RAG to security tasks. CyberRAG [ 20 ]presents an agentic framework for cyber attack classification, using iterative retrieval over domain-specific knowledge bases to reduce false positives and generate explanations. AgCyRAG [ 21 ] combines knowledge graph retrieval with vector search for security log anal-ysis, orchestrating specialized agents to integrate structured threat intelligence with security log data. Work on threat intelligence automation [ 16 ] uses RAG to extract Indicators of Compromise from CTI reports and generate regular expressions to assist security analysts in establishing SIEM correlation rules. GraphRAG-based approaches [ 22 ] leverage ontology-backed knowledge graphs con-structed from network logs to support LLM-driven query genera-tion for network security monitoring. These systems demonstrate RAG’s applicability to security data, primarily focusing on attack classification and threat intelligence extraction. Our work applies RAG to a different task: answering structured security questions over heterogeneous log sources (Suricata alerts, Zeek connections, Windows events). This requires smart chunk-ing strategies tailored to security data and retrieval that supports reasoning about attack progression across multiple log sources. 

## 3 Problem Formulation 

We formalize security incident analysis as a question-answering task and define the evaluation metrics used throughout our experi-ments.

Concrete Questionnaire. Throughout our experiments, we use a questionnaire of M = 5 questions targeting the essential information incident responders need to scope and contain an attack: (1) What is the IP address of the potentially infected internal host in the LAN? ( 𝜏 = ip )(2) What is the hostname of the potentially infected machine in the LAN? ( 𝜏 = hostname )(3) What is the Windows user account name of the potentially infected machine in the LAN? ( 𝜏 = domain_set )(5) What are the suspicious external IP addresses contacted, which might be involved in command-and-control (C2) communication? ( 𝜏 = ip_set )Questions 1—3 identify affected assets, question 4 determines the initial compromise vector, and question 5 pinpoints command-and-control infrastructure. These represent core forensic questions com-mon across incident types. The framework naturally extends to additional questions (e.g., "What MITRE ATT&CK techniques were used?" or "What files were dropped?") without modification to the underlying architecture. Beyond forensic identification, we also evaluate attack recon-struction: deducing the attack path a malicious actor took over a given timeframe. For this task, we pose a compound question ("What did the attacker do? What defenses should be implemented?") and evaluate responses using precision (fraction of reported attack steps matching reference) and recall (fraction of advised remedi-ations matching reference). This formulation unifies both modes within the question-answering framework, although attack recre-ation requires richer evaluation against known attack timelines (Section 7).

## 4 System Architecture 

Our architecture comprises two components: Security Context Ex-traction, which reduces raw logs to security-relevant information through targeted queries, and RAG-LLM Analysis, which retrieves relevant context and prompts an LLM to answer forensic questions. Figure 1 illustrates the overall architecture. 

## 4.1 Design Principles 

Five principles guided our architectural decisions. 

1) Filter before analysis. LLMs are too slow to process large volumes of raw logs and perform poorly on noisy data. We first extract security-relevant information using targeted IOC queries, allowing the system to scale to tractable log volumes and generate interpretable reports. 

2) Embed once, query many times. Extracted data is seman-tically embedded into a vector database and cached to disk. The index is built once per dataset and can be queried repeatedly with different questions, LLM providers, or prompt strategies without reprocessing. 

3) Periodic analysis. Rather than invoking the LLM on every event, the system performs analysis at defined intervals, batching related activity into coherent time windows. 

4) Provider-agnostic design. LLM interactions are abstracted behind a common interface, enabling substitution of providers to balance cost, latency, and privacy requirements. 

5) Extensible IOC query library. Detection queries are main-tained as a separate library with associated MITRE ATT&CK tech-niques, allowing extension without modifying the core pipeline. 

## 4.2 Security Context Extraction 

The system integrates with existing network monitoring infras-tructure through standard APIs. Our implementation uses Security Onion, an open-source platform that aggregates network traffic using Suricata and Zeek, communicating through ElasticSearch REST API calls. The architecture generalizes to any SIEM exposing a query API. Security Context Extraction relies on a library of IOC extraction queries, each targeting a specific activity type. The current query library was developed iteratively for the eval-uated attack types using domain expertise and LLM assistance, validating each query against known attack data; extending cover-age to novel attack patterns requires adding corresponding queries, though the modular design facilitates this extension. ## 4.3 RAG-LLM Analysis

Given a set of secu-rity questions, it retrieves relevant context from the embedded document store and prompts an LLM to produce evidence-based findings. This design decouples index construction from analysis, enabling rapid experimentation with different questions, providers, and prompt strategies without reprocessing the source data.

Identification questions focus on indicators of compromise: infected hosts, compromised accounts, malicious domains, and command-and-control servers. Timeline questions focus on attack progression: what steps did the attacker take, in what order, and when. Reme-diation questions focus on defensive actions: what steps should be taken to contain the incident and prevent further damage. All ques-tion types use the same retrieval and prompting pipeline, with the LLM adapting its response format based on the question framing. 

Multi-LLM Support. Following design principle number 4, the system abstracts LLM interactions behind a common interface, allowing different providers to be substituted without modifying the analysis pipeline. Currently supported providers include Anthropic, OpenAI, and DeepSeek as cloud APIs, and Llama and Cisco as locally-deployed options.

## 5 Experimental Overview 

We evaluate the system on two complementary scenarios: malware traffic analysis, which tests forensic question-answering, and Active Directory (AD) attack analysis, which tests attack reconstruction and defense recommendation. This section presents our research questions, shared experimental setup, and evaluation metrics; Sec-tions 6 and 7 present results for each scenario. 

Research Questions. We explore the following questions: 

RQ1 (Feasibility) Can a RAG-based system achieve accept-able performance on security incident analysis? 

RQ2 (Provider Comparison) How do different LLM providers compare in performance, latency, and cost? 

RQ3 (Context Sensitivity) How does the number of retrieved chunks 𝑘 affect performance? 

RQ4 (Reliability) How consistent is performance across dif-ferent attack types? 

RQ5 (RAG Necessity) Is RAG preprocessing essential, or can LLMs analyze raw security logs directly? 

RQ6 (Practical Utility) Do system-generated defense rec-ommendations align with expert-defined actions? RQ1–RQ5 are addressed in the malware traffic evaluation (Sec-tion 6), which compares eight LLM configurations across four attack scenarios and evaluates a no-RAG baseline. RQ1 and RQ6 are ad-dressed in the Active Directory evaluation (Section 7), which tests attack detection and defense recommendation on a multi-stage intrusion. 

LLM Providers and Models. We evaluate eight configurations spanning cloud APIs and local deployment: 

• Claude Sonnet 4 (Anthropic): Cloud API 

• DeepSeek V3 (DeepSeek): Cloud API 

• GPT-4, GPT-4o, GPT-5-mini, GPT-5.2 (OpenAI): Cloud API 

• Llama 3.1:70b (via Ollama): Local GPU deployment 

• Cisco Foundation-Sec-8B : Local GPU deployment, security-specialized The malware evaluation uses all providers; the AD evaluation uses Llama 3.1 to demonstrate local deployment feasibility. 

RAG Configuration. The RAG pipeline uses the following pa-rameters: 

• Embedding model: all-mpnet-base-v2 (sentence trans-former) 

• Vector store: FAISS with cosine similarity 

• Retrieved chunks: 𝑘 = 7 per question (default); Section 6.3 examines sensitivity to 𝑘 

• LLM parameters: temperature = 0.1, max output tokens =

1500 (2000 for Cisco to avoid truncation) Each chunk contains a complete aggregation result rather than ar-bitrary text segments, preserving semantic coherence for retrieval. 

Evaluation Metrics. We evaluate against reference sets derived from published scenario solutions and manual log analysis. Let 𝑅 

denote the reference set and 𝐴 denote the model’s answer set. We compute: Precision = |𝐴 ∩ 𝑅 ||𝐴 | , Recall = |𝐴 ∩ 𝑅 ||𝑅 |

For single-value questions (e.g., infected host IP), precision equals recall and scoring is binary. For set-valued questions (e.g., C2 indi-cators), models receive partial credit proportional to the fraction of reference indicators found. The malware evaluation primarily reports recall, as the reference answers capture confirmed malicious indicators from published solutions rather than an exhaustive enumeration of all suspicious activity. Reference answers were validated by two analysts who independently reviewed the raw data using Wireshark and the Security Onion Kibana dashboard. Models may identify additional indicators (Appendix I) that are legitimately suspicious but absent from the reference set. Appendix H provides detailed precision-recall breakdowns showing that several reported indicators not in the reference set exhibit genuinely suspicious characteristics. The AD evaluation reports both precision and recall, as its reference timeline was manually extracted through exhaustive log review and cross-checked by two analysts. 

## 6 Malware Traffic Analysis 

Data. We evaluate on four malware traffic scenarios from malware-traffic-analysis.net [ 10 ] and Unit 42 [ 24 ], each representing a dis-tinct attack pattern: 

• Fake Authenticator (Jan 2025): User directed to fraudu-lent Google Authenticator download page, leading to multi-server C2 communication. 

• NetSupport RAT (Nov 2024): Legitimate website injected with malicious scripts delivering fake browser updates and a Remote Access Trojan. 

• Koi Stealer (Sep 2024): Information-stealing malware with multiple C2 endpoints and regular check-in behavior.

## 6.2 Reasoning Pattern Analysis 

Beyond recall metrics, models exhibit qualitatively different reason-ing strategies. In Appendix G, we compare responses to the C2 detection ques-tion (Q5) on the Koi Stealer scenario, where the reference C2 server is 79.124.78.197 . DeepSeek identifies only the confirmed C2 server, explicitly rejecting domain-resolved IPs as insufficiently corroborated. Cisco provides tiered findings—the primary C2 plus secondary suspicious IPs with context—and notes operational con-siderations like domain fronting. GPT-5.2 lists seven IPs without prioritization, including the internal victim host ( 172.17.0.99 ) as a “suspicious external IP”—a category error revealing shallow pattern matching rather than analytical reasoning. These patterns reflect a precision-coverage trade-off: DeepSeek minimizes false positives but risks missing secondary infrastruc-ture; Cisco optimizes for SOC workflows where comprehensive threat visibility is prioritized; GPT-5.2’s undifferentiated enumer-ation increases analyst burden. Appendix H provides detailed precision-recall analysis for each scenario. For production de-ployment, evidence-weighted reasoning (distinguishing confirmed threats from suspicious activity) is preferable to exhaustive listing.

## 7.2 Multi-Step Attack Reconstruction 

We examine how the system reconstructed an ESC1 certificate template abuse attack across five discrete events: certificate enu-meration, certificate request with SAN:Administrator , certificate issuance with Subject:CN=<user> , Kerberos TGT as Administra-tor, and service ticket for krbtgt . Each event was captured by an independent query targeting specific Windows event codes (4662, 4886, 4887, 4768, 4769). Given these events as separate chunks, the system produced: 

> Red enumerated certificates, then requested a certificate with SAN=Administrator. The certificate was issued, allow-ing impersonation of the Administrator account. Red then obtained a Kerberos TGT as Administrator and requested a service ticket for krbtgt, achieving domain-wide access.

The exploit mechanism required connecting informa-tion across chunks. The certificate request chunk contained 

SAN:upn=Administrator , while the certificate issued chunk showed Subject:CN=<user> . The LLM identified this Subject/SAN mismatch as the privilege escalation vector, then linked it to subsequent Kerberos events. 

Finding 5: The system distinguishes attack from benign ac-tivity by recognizing that individually legitimate operations (certificate issuance, Kerberos authentication) constitute ex-ploitation when they enable privilege escalation, demonstrat-ing contextual interpretation beyond pattern matching ( RQ1 ).

## 7.4 Operational Feasibility 

The system completes analysis in approximately 61 seconds per 5-minute window: 466ms for query execution (9 queries) and under 60 seconds for LLM analysis using locally-deployed Llama 3.1:70b. This represents 20% of the window duration, leaving time for analyst review and response actions before the next analysis cycle. This AD scenario represents a fast-moving attack with core exploitation compressed into 60 seconds. The 5-minute window suited this pace, but attack timescales vary widely: ransomware may complete encryption in under 5 minutes, while APT campaigns spread lateral movement over days. Window size must be tuned to threat characteristics. More fundamentally, analysis latency should be measured against response latency rather than attack speed. Defensive actions may require several minutes to execute, and co-ordinated responses involve human decision-making and approval workflows that typically dominate total incident response time. The system’s sub-minute analysis latency makes detection a small fraction of end-to-end response. 

## 8 Conclusion 

We presented a retrieval-augmented generation system for security incident analysis that combines targeted query-based filtering with LLM semantic reasoning. Evaluation across eight LLM models on malware and Active Directory scenarios demonstrates that RAG pre-processing is essential (no-RAG baselines miss all attack infrastruc-ture), provider selection dominates retrieval tuning, and DeepSeek V3 achieves Claude-equivalent accuracy (100% recall) at 15 × lower cost. Notably, security-specialized models (Cisco Foundation-Sec-8B) do not outperform general-purpose LLMs, suggesting that rea-soning capability matters more than domain-specific pretraining for forensic analysis. Our system’s key capability is cross-event Cadet et al. 

semantic reasoning: distinguishing coordinated exploitation from individually legitimate operations by correlating evidence across discrete forensic events. Our evaluation covers four malware sce-narios and one Active Directory attack; while results are consistent across these cases, validation on a broader set of attack types and larger-scale deployments remains future work.

## I IcedID Dataset Complexity 

Here we present some aspects that make the IcedID malware particularly challenging, and how our system uncovers attack indicators. 

Multi-Host Environment with Mixed Traffic. The capture contains traffic from three Windows hosts in an Active Directory environment, with only one confirmed compromise. DESKTOP-SFF9LJF (10.4.19.136) follows the documented IcedID chain: Firebase-hosted installer, HTTP binary retrieval, persistent HTTPS C2 beaconing, and BackConnect activity. Other hosts in the environment (including DESKTOP-RETP4BU at 10.4.19.138) generate background traffic that includes DNS queries and web requests unrelated to the IcedID infection. This multi-host traffic creates analytical noise, as there is no structural boundary separating the infected host’s footprint from routine AD activity. 

Volume as a Misleading Signal. Domain join sequences and routine AD operations from uninfected hosts produce heavy Kerberos and LDAP bursts that are volumetrically louder than the infected host’s quieter but diagnostically significant IcedID C2 beaconing, making raw traffic volume an unreliable primary indicator in this capture. 

Reference Set Scope. The Unit 42 reference answers cover six IOCs for a single host. During evaluation, our system surfaced additional indicators confirmed valid upon manual investigation: 

• Additional C2 domain. Our system cited spakernakurs.com resolving to 104.168.53.18 , identical to askamoshopsi.com —and absent from the Unit 42 writeup. Manual lookup confirmed the domain resolves to shared IcedID infrastructure, consistent with a rotation or replacement C2 domain. 

• Beaconing regularity. Our system’s classification of skansnekssky.com as persistent C2 activity prompted tshark verification, confirming ∼302-second intervals with sub-second jitter across 69+ sessions, a pattern unquantified in the reference answers. 

Query Adaptation. The IcedID installer delivery via Firebase Storage ( firebasestorage.googleapis.com ) required query modification: the standard *googleapis.com* exclusion for noise reduction would suppress the entire infection vector. We replaced broad wildcards with surgical per-subdomain exclusions and added two aggregations: (1) direct-IP TLS connections, recovering BackConnect C2 at 193.149.176.100, and (2) HTTP requests to raw IP hosts, recovering the redirect server at 80.77.25.175. This adaptation illustrates how attackers exploit common filtering heuristics and the need for dataset-aware query refinement.

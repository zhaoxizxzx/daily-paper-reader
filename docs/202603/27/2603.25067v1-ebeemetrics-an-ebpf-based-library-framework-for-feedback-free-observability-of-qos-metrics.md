## 摘要
许多系统管理运行时（SMR），例如资源管理和功耗管理技术，依赖于服务质量（QoS）指标（如尾部延迟或吞吐量）作为反馈。这些 QoS 指标通常既无法通过硬件性能计数器观测，也无法在操作系统内核中直接观测。这在对应用程序进行插桩以及将 QoS 性能指标反馈与多种管理运行时集成时，引入了复杂性和开销。为了弥补这一差距，我们推出了 eBeeMetrics，这是一个基于 eBPF 的库框架，能够准确观测仅由 eBPF 可观测事件（如系统调用）推导出的应用级指标。eBeeMetrics 可作为即插即用的替代方案，将系统管理运行时与 QoS 指标反馈报告解耦，或者补充现有的 QoS 指标以更好地识别服务器端动态。eBeeMetrics 在各种延迟敏感型工作负载中，与实际测量的吞吐量和延迟指标表现出强相关性。eBeeMetrics 工具已开源，源代码可在以下网址获取：https://github.com/Ibnathism/eBeeMetrics。

## Abstract
Many system management runtimes (SMRs), such as resource management and power management techniques, rely on quality-of-service (QoS) metrics, such as tail latency or throughput, as feedback. These QoS metrics are generally neither observable with hardware performance counters nor directly observable within the OS kernel. This introduces complexity and overhead in instrumenting the application and integrating QoS performance metric feedback with many management runtimes. To bridge this gap, we introduced eBeeMetrics, an eBPF-based library framework to accurately observe application-level metrics derived from only eBPF-observable events, such as system calls. eBeeMetrics can be used as a drop-in replacement to decouple system management runtimes from QoS metric feedback reporting, or can supplement existing QoS metrics to better identify server-side dynamics. eBeeMetrics achieves a strong correlation with real-world measured throughput and latency metrics across various latency-sensitive workloads. The eBeeMetrics tool is open-source; the source code is available at: https://github.com/Ibnathism/eBeeMetrics.

### eBeeMetrics：基于 eBPF 的 QoS 指标无反馈可观测性库框架总结

#### 1. 论文的核心问题与整体含义（研究动机和背景）
现代数据中心依赖系统管理运行时（SMR）进行资源调度、功耗管理和性能调试。这些管理决策通常需要服务质量（QoS）指标（如尾部延迟、吞吐量）作为反馈。然而，QoS 指标在操作系统内核或硬件计数器中不可直接观测，传统方法通常需要：
*   **应用插桩**：修改代码以报告指标，增加了开发复杂性和运行时开销。
*   **侵入式反馈**：通过 `sysfs` 等机制传递指标，系统调用开销大，难以实现实时反馈。
*   **离线分析**：现有研究多侧重于离线分析 eBPF 追踪记录，无法满足实时控制回路的需求。

论文提出了 **eBeeMetrics**，旨在通过 eBPF 技术，在不修改应用程序的前提下，从内核事件（如系统调用）中实时、准确地推导出应用级 QoS 指标。

#### 2. 论文提出的方法论
eBeeMetrics 的核心思想是**请求去混淆（Request Disambiguation）**，即在并发环境下准确识别单个请求的开始和结束边界。

*   **核心架构**：
    *   **内核空间（eBPF 程序）**：使用 `kprobes`（内核探针）或 `uprobes`（用户空间探针）拦截特定事件。
    *   **数据传输**：利用 eBPF 环形缓冲区（Ring Buffer）将请求级元数据（时间戳、标识符）异步传输至用户空间，以最小化开销。
    *   **用户空间（库框架）**：聚合原始数据，计算吞吐量、平均延迟和尾部延迟（如 99th 百分位），并通过 API 提供给 SMR。
*   **关键技术细节（协议感知）**：
    *   **HTTP/1.1**：通过拦截 `accept4` 获取文件描述符（FD）作为请求标识符，匹配对应的 `close` 系统调用来确定请求生命周期。
    *   **gRPC/HTTP/2**：由于多路复用，系统调用层难以区分请求。eBeeMetrics 使用 `uprobes` 拦截 gRPC 库函数（如 `chttp2_stream` 构造函数和元数据完成回调），利用流 ID（Stream ID）进行匹配。
    *   **持久连接（如 Memcached）**：针对不频繁关闭连接的应用，通过拦截 `read`（请求开始）和 `sendmsg`（响应结束）来估算延迟。

*   **可观测性增强**：eBeeMetrics 仅测量服务器端处理时间，结合客户端指标可以帮助识别性能瓶颈是在网络端还是服务器端。
